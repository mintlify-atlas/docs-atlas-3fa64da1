## Overview

The registration endpoint creates a new user account in the GreenGuard system. It accepts user profile information and creates a record in the database with default values for points and activation status.

## Endpoint

```
POST /auth/register
```

## Request Body

<ParamField body="nomUsu" type="string" required>
  User's first name
  
  Example: `Juan`
</ParamField>

<ParamField body="apePatUsu" type="string" required>
  User's paternal surname
  
  Example: `Perez`
</ParamField>

<ParamField body="apeMatUsu" type="string" required>
  User's maternal surname
  
  Example: `Garcia`
</ParamField>

<ParamField body="documentoUsu" type="string" required>
  User's identification document number. Must be unique in the system.
  
  Example: `12345678`
</ParamField>

<ParamField body="correoUsu" type="string" required>
  User's email address. Must be unique and valid format.
  
  Example: `juan.perez@example.com`
</ParamField>

<ParamField body="passwordUsu" type="string" required>
  User's password. Should meet minimum security requirements.
  
  Example: `SecurePassword123!`
</ParamField>

<ParamField body="telefonoUsu" type="string" required>
  User's phone number with country code
  
  Example: `+51987654321`
</ParamField>

<ParamField body="generoUsu" type="string" required>
  User's gender. Typically "M" (Male), "F" (Female), or other options.
  
  Example: `M`
</ParamField>

<ParamField body="imagenUSU" type="string">
  URL or path to user's profile image. Optional field.
  
  Example: `/images/profiles/default.jpg`
</ParamField>

<ParamField body="rol" type="object" required>
  User's role assignment. Contains role identifier.
  
  <Expandable title="rol properties">
    <ParamField body="idRol" type="integer" required>
      Role identifier (e.g., 1 for "User", 2 for "Admin")
      
      Example: `1`
    </ParamField>
  </Expandable>
</ParamField>

## Validation Rules

The registration endpoint enforces the following validation rules:

<AccordionGroup>
  <Accordion title="Email Validation">
    - Must be a valid email format
    - Must be unique in the system
    - Cannot be empty or null
  </Accordion>

  <Accordion title="Password Requirements">
    - Minimum length of 8 characters (recommended)
    - Should contain a mix of letters, numbers, and special characters
    - Passwords are encrypted before storage
  </Accordion>

  <Accordion title="Document Number">
    - Must be unique in the system
    - Cannot be empty or null
    - Format depends on regional requirements
  </Accordion>

  <Accordion title="Phone Number">
    - Should include country code
    - Must be a valid format
  </Accordion>

  <Accordion title="Role Assignment">
    - Role ID must exist in the `tb_rol` table
    - Typically defaults to standard "User" role (ID: 1)
  </Accordion>
</AccordionGroup>

## Default Values

When a user is registered, the following fields are automatically set:

| Field | Default Value | Description |
|-------|---------------|-------------|
| `registroUsu` | Current timestamp | Set when the account is created |
| `puntosUsu` | 0 | Users start with zero points |
| `activo` | true | New accounts are active by default |
| `fmcToken` | null | Set when user enables push notifications |
| `fmcTokenFecha` | null | Set when FCM token is first created |

## Database Storage

User data is stored in the `tb_usuario` table with the following column mappings:

- `id_usu` - Auto-generated unique identifier
- `nom_usu` - First name
- `ape_pat_usu` - Paternal surname
- `ape_mat_usu` - Maternal surname
- `documento_usu` - Document number
- `correo_usu` - Email address
- `password_usu` - Encrypted password
- `telefono_usu` - Phone number
- `genero_usu` - Gender
- `imagen_usu` - Profile image path
- `registro_usu` - Registration timestamp
- `puntos_usu` - Reward points
- `ROL` - Foreign key to `tb_rol` table
- `activo` - Active status flag
- `FCM_TOKEN` - Push notification token
- `FCM_TOKEN_FECHA` - Token creation date

## Security Considerations

<Warning>
  Always use HTTPS when transmitting registration data. Personal information and passwords must be encrypted in transit.
</Warning>

<Info>
  Best practices for registration:
  - Implement rate limiting to prevent spam registrations
  - Use email verification to confirm account ownership
  - Enforce strong password requirements
  - Sanitize all input to prevent SQL injection
  - Log registration attempts for security monitoring
</Info>

## Post-Registration Flow

After successful registration, typical next steps include:

1. **Email Verification**: Send a verification email to confirm the user's email address
2. **Welcome Message**: Display onboarding information or tutorial
3. **Profile Completion**: Prompt user to complete optional profile fields
4. **Push Notification Setup**: Request permission for notifications and set `fmcToken`

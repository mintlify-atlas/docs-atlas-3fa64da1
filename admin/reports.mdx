The report management system allows administrators and municipal staff to review, process, and resolve environmental incident reports submitted by users across the platform.

## Report Data Model

Environmental reports are stored in the `tb_reporte` table with comprehensive tracking:

### Report Fields

| Field | Type | Description |
|-------|------|-------------|
| `idReporte` | Integer | Unique report identifier (auto-generated) |
| `numReport` | String | Report number for reference |
| `usuario` | Usuario | User who submitted the report (FK) |
| `detalleRepo` | String | Detailed description of the incident |
| `imagenRepo` | String | Image URL/path of the incident |
| `estado` | EstadoReporte | Current report status (enum) |
| `latitud` | BigDecimal | Latitude coordinate |
| `longitud` | BigDecimal | Longitude coordinate |
| `tipoIncidente` | TipoIncidentes | Type of environmental incident (FK) |
| `tipoClasificacion` | TipoClasificacion | Classification category (FK) |
| `distrito` | Distrito | District/location (FK) |
| `repoRegistado` | LocalDateTime | Report submission timestamp |
| `repoResuelto` | LocalDateTime | Resolution timestamp |

## Report Status Workflow

Reports progress through defined states tracked in the `EstadoReporte` enum:

<Steps>
  <Step title="PE - Pendiente (Pending)">
    **Initial Status**
    
    New reports start in pending state awaiting review:
    - Recently submitted by users
    - Not yet assigned or reviewed
    - Awaiting administrator triage
    
    ```java
    // Initial state
    PE, //pendiente
    ```
  </Step>
  
  <Step title="EP - En Proceso (In Progress)">
    **Active Processing**
    
    Report is being actively worked on:
    - Assigned to municipal staff
    - Investigation underway
    - Resolution in progress
    
    ```java
    EP, //En Proceso
    ```
  </Step>
  
  <Step title="RE - Resuelto (Resolved)">
    **Completed**
    
    Report has been successfully resolved:
    - Issue addressed and fixed
    - Resolution timestamp recorded
    - User can be notified
    - Points awarded to reporter
    
    ```java
    RE, //Resuelto
    ```
  </Step>
  
  <Step title="CA - Cancelado (Canceled)">
    **Canceled/Invalid**
    
    Report was canceled or deemed invalid:
    - Duplicate report
    - Insufficient information
    - Out of jurisdiction
    - False report
    
    ```java
    CA //Cancelado
    ```
  </Step>
</Steps>

<Tabs>

  <Tab title="Pending Reports">

    SELECT * FROM tb_reporte

  </Tab>
</Tabs>

### Processing New Reports

When a new report is submitted, follow this workflow:

<Steps>
  <Step title="Review Report Details">
    Examine the submitted information:
    
    **Key Information**
    - Reporter: `usuario.nomUsu` (user full name)
    - Description: `detalleRepo`
    - Location: `latitud`, `longitud`, `distrito`
    - Image: `imagenRepo` (visual evidence)
    - Type: `tipoIncidente` (incident category)
    - Classification: `tipoClasificacion`
    - Submission date: `repoRegistado`
  </Step>
  
  <Step title="Validate Report">
    Verify the report is legitimate:
    
    <Check>Description is clear and detailed</Check>
    <Check>Location coordinates are valid</Check>
    <Check>Image shows the reported incident</Check>
    <Check>Incident type matches the description</Check>
    <Check>Not a duplicate of existing report</Check>
  </Step>
  
  <Step title="Assign Classification">
    Categorize using `tipoClasificacion`:
    - Urgency level
    - Resource requirements
    - Responsible department
  </Step>
  
  <Step title="Update Status">
    Move to appropriate status:
    
    ```java
    // Update to In Progress
    reporte.setEstado(EstadoReporte.EP);
    ```
    
    Or cancel if invalid:
    
    ```java
    // Cancel invalid report
    reporte.setEstado(EstadoReporte.CA);
    ```
  </Step>
</Steps>

### Resolving Reports

To mark a report as resolved:

<Steps>
  <Step title="Verify Resolution">
    Confirm the incident has been addressed:
    - Physical inspection completed
    - Issue remediated
    - Documentation collected
  </Step>
  
  <Step title="Update Status to Resolved">
    Change status and record timestamp:
    
    ```java
    reporte.setEstado(EstadoReporte.RE);
    reporte.setRepoResuelto(LocalDateTime.now());
    ```
  </Step>
  
  <Step title="Award Points to Reporter">
    Credit the user who submitted the report:
    
    ```java
    Usuario reporter = reporte.getUsuario();
    int currentPoints = reporter.getPuntosUsu();
    reporter.setPuntosUsu(currentPoints + pointsEarned);
    ```
    
    <Note>
    Points awarded vary based on incident type and resolution complexity.
    </Note>
  </Step>
  
  <Step title="Send Notification">
    Notify the user of resolution:
    - Use FCM token for push notification
    - Include resolution details
    - Show points awarded
  </Step>
</Steps>

### Reporting Queries

```sql
-- Resolution time analysis
SELECT 
  AVG(TIMESTAMPDIFF(HOUR, repo_registado, repo_resuelto)) as avg_hours
FROM tb_reporte
WHERE estado = 'RE' AND repo_resuelto IS NOT NULL;

-- Reports by status
SELECT estado, COUNT(*) as count
FROM tb_reporte
GROUP BY estado;

-- Most active reporters
SELECT u.nom_usu, u.ape_pat_usu, COUNT(r.id_reporte) as report_count
FROM tb_usuario u
JOIN tb_reporte r ON u.id_usu = r.id_usu
GROUP BY u.id_usu
ORDER BY report_count DESC
LIMIT 10;
```

## Report Timeline Tracking

### Registration and Resolution Timestamps

```java
@Column(name="repo_registrado")
private LocalDateTime repoRegistado;

@Column(name="repo_resuelto")
private LocalDateTime repoResuelto;
```

Track the complete report lifecycle:
- **Submission**: `repoRegistado` captures initial report time
- **Resolution**: `repoResuelto` records completion time
- **Duration**: Calculate time difference for metrics

<Note>
The `repoResuelto` field is only populated when status changes to `RE` (Resuelto).
</Note>

## Integration with Notifications

### Notification Triggers

Reports trigger notifications at key stages:

1. **Report Submitted** - Confirm receipt to user
2. **Status Changed to EP** - Notify user work has started
3. **Report Resolved** - Notify completion and points awarded
4. **Report Canceled** - Explain cancellation reason

Use the NotificacionController for sending:

```java
@RestController
@RequestMapping("/notificacion")
public class NotificacionController {
    @Autowired
    NotificacionService notificacionService;
}
```

## Best Practices

<Warning>
Follow these guidelines for effective report management:
</Warning>

### Response Times

- **Pending → In Progress**: Within 24 hours
- **In Progress → Resolved**: Based on incident type (1-30 days)
- **High Priority**: Immediate response required

### Quality Control

- Verify all field data before status changes
- Require evidence for resolutions
- Document cancellation reasons
- Maintain audit trail

### User Communication

- Provide status update notifications
- Explain cancellation decisions
- Confirm resolutions with evidence
- Thank users for environmental stewardship
